## Architectural Decision Records

ADRs record significant technology choices with their status, rationale, and consequences. Each ADR is a short markdown file. All current ADRs have status **Accepted**.

| ADR | Decision | Key rationale |
|---|---|---|
| [ADR-001](adr/ADR-001-monorepo-architecture.md) | Single monorepo (uv + pnpm workspaces) | Simplifies dependency management, cross-service refactoring, and air-gap bundle creation |
| [ADR-002](adr/ADR-002-spiffworkflow-for-bpmn-and-dmn-execution.md) | SpiffWorkflow for BPMN/DMN | Pure Python, LGPL-2.1 (dynamic link — no propagation), serialisable state, air-gap compatible |
| [ADR-003](adr/ADR-003-keycloak-for-identity-and-access-management.md) | Keycloak for IAM | OIDC + SAML + LDAP + MFA + service accounts; fully on-premises; FedRAMP-aligned |
| [ADR-004](adr/ADR-004-minio-for-object-storage.md) | MinIO for object storage | S3-compatible, on-premises, WORM object locking for retention; AGPL is acceptable for internal use |
| [ADR-005](adr/ADR-005-valkey-(redis-fork)-for-messaging-and-caching.md) | Valkey over Redis | Redis re-licensed to SSPL in 2024; Valkey is BSD-3-Clause, drop-in compatible, Linux Foundation governed |

---

## ADR Format

Each ADR follows this structure:

```markdown
# ADR-NNN: Title

**Status:** Proposed | Accepted | Deprecated | Superseded by ADR-NNN

**Date:** YYYY-MM-DD

## Decision
What was decided.

## Consequences
Positive and negative outcomes.
```

---

## Writing a New ADR

1. Create `docs/adr/ADR-{NNN}-{short-title}.md` (next sequential number).
2. Set status to `Proposed` until the decision is finalised.
3. Reference the ADR from the relevant service or package README.
4. If superseding an existing ADR, update the old ADR's status to `Superseded by ADR-NNN`.

## Compliance (`docs/compliance/`)

This directory will hold the NIST 800-53 / FedRAMP Moderate control mapping — a spreadsheet-style document cross-referencing each control family to the GXP components that implement it.

Automated evidence for the AU control family is generated by `tests/compliance/test_au_controls.py`. Controls that require manual evidence (physical access, personnel security, etc.) are out of scope for automated tests and will be documented here.

Planned control families to document:

| Family | Description | Primary GXP components |
|---|---|---|
| AU | Audit and Accountability | audit-service, py-shared/audit |
| AC | Access Control | Keycloak, py-shared/auth/dependencies |
| IA | Identification and Authentication | Keycloak (MFA, PKCE, short-lived tokens) |
| SC | System and Communications Protection | Traefik (TLS), Keycloak (encrypted tokens) |
| SI | System and Information Integrity | ClamAV (document scanning), Bandit/Semgrep (SAST) |
| CM | Configuration Management | Alembic migrations, Docker image pinning |
